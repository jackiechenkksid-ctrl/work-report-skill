# 工作汇报生成器

按照 **SMART 原则**帮用户生成结构化的日报或周报。核心目标：**不是记流水账，而是写出有数据、有逻辑、有结论的高质量报告。**

---

## SMART 原则速查

AI 在生成报告的每个环节，必须用这 5 个维度自检：

| 维度 | 含义 | 自检问题 |
|------|------|----------|
| **S**pecific（具体） | 清晰明确，不模糊 | 别人一看就懂你做了什么吗？ |
| **M**easurable（可衡量） | 有数字、有进度 | 有量化数据佐证吗？进展百分比？ |
| **A**chievable（可达成） | 现实可行，有步骤 | 资源和时间支撑吗？下一步怎么拆？ |
| **R**elevant（相关性） | 对齐大目标 | 这件事和团队/公司目标的关系是什么？ |
| **T**ime-bound（时限） | 有截止时间 | 什么时候完成？有明确节点吗？ |

**写报告时的 SMAT 反例对照：**

> ❌ «本周推进了客户增长工作»
>
> ✅ «本周新签付费客户 4 个（目标 5 个，完成率 80%），拜访 12 家潜客，发出方案 8 份。未达标原因：2 家客户推迟到下周签约。下周目标：新签 5 个，周三前完成 10 家触达。」

## Phase 1: 确认报告参数

开始前，确认以下信息（用户未提供则主动询问）：

| 参数 | 说明 | 示例 |
|------|------|------|
| 报告类型 | 日报 or 周报 | 日报 |
| 日期范围 | 日报默认今天，周报默认本周一至周五 | 2026-06-03 / 本周 |
| 工作数据 | 工作内容从哪来 | 粘贴 / 系统URL+账号 / CSV文件 / 飞书 / 口头描述 |
| 报告格式 | 使用默认 SMART 模板 / 用户自定义模板 | 默认 / 自定义 |

**只需问这一次**，不要反复追问。

### 模板自定义

如果用户希望用自己的格式而非默认 SMART 模板，支持两种方式：

**方式 A：用户粘贴历史报告**

用户从之前的日报/周报中复制一篇，AI 自动提取格式：

```text
用户: "以后日报按这个格式写：[粘贴一篇之前的日报]"

AI 解析：
- 识别所有章节标题（如「今日工作」「明日计划」「问题」等）
- 识别格式风格（表格 vs 列表、emoji 使用习惯、落款方式）
- 识别内容密度（字数、条目数、结构化程度）
- 提取后展示给用户确认
```

**方式 B：用户口头描述**

用户直接说想要的格式：
- "我要那种简洁的，就 3 块：做了什么 / 遇到什么问题 / 明天干嘛"
- "按 OKR 格式写，每个目标下面跟 KR 完成度"

**模板持久化：** 提取后的模板保存为 `templates/custom-daily.md` 和 `templates/custom-weekly.md`，后续生成报告时优先使用自定义模板。没有自定义模板时才用默认 SMART 模板。

**模板优先级：**
1. 用户本次对话中指定的格式
2. `templates/` 目录下保存的自定义模板
3. 默认 SMART 模板（Phase 4）

## Phase 2: 收集工作数据

根据用户选择的方式收集原始数据：

### 2A. 用户口述/粘贴

用户直接描述今天做了什么，或粘贴碎片化笔记时：

1. 接受任意格式：纯文本、列表、聊天记录片段
2. 提取关键信息：任务名称、完成状态、时间花费、产出物
3. 用 SMART 框架整理后回传给用户确认：
   ```
   我整理了一下，请确认：
   - 完成了 A 功能开发（代码 300 行，已提 PR）
   - 参加了产品评审会（14:00-15:00，产出 3 条需求变更）
   - B 项目排期延期了 2 天（原因是设计稿未交付）
   还有遗漏吗？
   ```

### 2B. 自动登录公司系统拉取

用户提供公司内部系统 URL 和账号密码时，使用脚本自动登录并拉取数据。

**Step 1：检查是否有专用脚本**

查看 `scripts/` 目录下是否有对应系统的脚本（如 `scripts/login_and_fetch.py`）。

**Step 2：首次使用 — 帮用户配置**

如果脚本存在但用户从未配置过，询问用户的登录凭据：

```
要自动拉取考勤数据，需要你的系统账号。
建议用环境变量存储（更安全），或直接传入。
你需要提供：
- 用户名/手机号
- 密码
```

**Step 3：运行脚本拉取数据**

```bash
# 单日查询（默认今天）
python3 scripts/login_and_fetch.py <username> <password> --date 20260603

# 日期范围查询（周报）
python3 scripts/login_and_fetch.py <username> <password> --range 20260601-20260605
```

脚本输出 JSON 格式的考勤数据，AI 解析后进入 Phase 3 加工。

**Step 4：脚本不匹配时 → API 自动探测协议**

如果用户系统没有现成脚本，AI 必须按以下流程**自动逆向工程**系统 API。这是从实战中沉淀的方法论。

**4.1 页面入口探测**

```
1. WebFetch 抓首页 → 如果只返回几个字（SPA），进入 4.2
2. 如果 WebFetch 返回完整内容 → 直接提取数据，跳过后续步骤
```

**4.2 下载 HTML 源码找 JS 包**

```bash
curl -sL <首页URL> | grep -oE 'src="[^"]*\.js[^"]*"'
# 重点找: /static/js/app.xxx.js, /static/js/chunk-vendors.xxx.js
```

**4.3 从 JS 中提取 API 配置**

```bash
# 下载 app.js，搜索关键配置
curl -sL <app.js URL> | tr ',' '\n' | grep -iE '(baseURL|base_url|login|api|auth|token)'

# 下载 chunk-vendors.js，搜索
curl -sL <vendors.js URL> | tr ',' '\n' | grep -iE '(baseURL|login|oauth|auth)'
```

**4.4 定位 OAuth2 登录接口**

```bash
# 1. 确认 baseURL（如 "/api"）
# 2. 找登录端点（如 "/oauth/token"）
# 3. 提取 client_id 和 client_secret
python3 -c "
import re
# 搜 JS 中的 client 凭证
re.findall(r'client_id:\"([^\"]+)\"', content)
re.findall(r'client_secret:\"([^\"]+)\"', content)
# 找 Basic Auth 构造逻辑
re.findall(r'Authorization.*Basic.*btoa', content)
"
```

**4.5 验证登录接口**

```python
import requests, base64

# 用提取的 client_id:client_secret 构造 Basic Auth
auth = base64.b64encode(b'client_id:client_secret').decode()

# POST OAuth2 password grant
resp = requests.post(f'{BASE_URL}/api/oauth/token', data={
    'grant_type': 'password',
    'username': username,
    'password': password,
    'scope': 'server',
}, headers={'Authorization': f'Basic {auth}'})

# 期望：code=000 + access_token，或 code=001 + 用户名密码错误
# 只要不是 404，说明接口正确
```

**4.6 探测数据接口**

拿到 token 后，从 JS 中提取的 URL 列表里筛选可能的查询接口：

```bash
# 从 JS 搜所有 url 定义
python3 -c "
import re
urls = re.findall(r'url:\"([^\"]+)\"', js_content)
# 筛选数据查询关键词
for u in urls:
    if any(kw in u for kw in ['checkin','record','list','query','track','statistic']):
        print(u)
"
```

然后用 token 依次测试，确认哪个路径返回数据。

**4.7 探测回填接口（可选）**

同上方法，筛选关键词 `statement`、`daily`、`week`、`save`、`update`。

先发空 body 看 400 响应中的必填字段提示：
```python
resp = requests.post(f'{BASE_URL}/api/statement/week', json={}, headers=headers)
# 从错误消息提取必填字段: "startDate, must not be null"
```

再尝试不同字段组合直到返回 code=000。

**4.8 生成脚本**

接口验证通过后，参照 `scripts/login_and_fetch.py` 模板生成对应系统的脚本，保存到 `scripts/` 目录。

> **核心原则：用 404 vs 500 vs 200 区分接口状态**
> - 404 → 路径不存在，换路径
> - 500 → 路径存在但参数不对，调整参数
> - 200 + code=001 → 业务逻辑拦截（如"已提交过"），接口正确
> - 200 + code=000 → 成功，记录字段结构

> **🔒 安全规则：**
> - 脚本中**不硬编码**用户的用户名和密码
> - 建议用户用环境变量：`export WORK_SYSTEM_USER=xxx`、`export WORK_SYSTEM_PASS=xxx`
> - 密码不要在聊天中明文显示

### 2C. CSV/Excel 文件

1. 先 `Read` 文件内容查看结构
2. 自动识别列名，提取工作项、进度、时间等字段
3. 如果文件较大（>50行），只读取目标日期范围的数据

### 2D. 飞书考勤/日历/任务（可选集成）

如果用户明确说使用飞书系统：

```bash
# 考勤打卡记录
lark-cli attendance user_tasks query \
  --data '{"user_ids":[],"check_date_from":<yyyyMMdd>,"check_date_to":<yyyyMMdd>}' \
  --params '{"employee_type":"employee_no"}'

# 日程
lark-cli calendar +agenda --start "<ISO8601>" --end "<ISO8601>"

# 未完成待办
lark-cli task +get-my-tasks --complete=false
```

> 使用前需读取 [`lark-shared`](../lark-shared/SKILL.md) 完成认证。

## Phase 3: 挖掘 SMART 要素

原始数据收集完后，AI 需要主动做一层"SMART 加工"——把模糊描述转化为 SMART 表达：

### 对每条工作项进行 SMART 补全

如果用户说"做了 XX 功能"：
- **S**: 追问「具体做了哪些模块/接口/页面？」
- **M**: 追问「代码量？提测覆盖多少 case？性能提升多少？」
- **A**: 追问「有没有遇到阻碍？怎么解决的？」
- **R**: 追问「这个功能服务于哪个业务目标？」
- **T**: 追问「什么时候上线的？还有遗留工作吗？预计什么时候收尾？」

如果用户无法提供量化数据，AI 应基于上下文合理推断并用「约/预计」标注，**不能编造精确数字**。

## Phase 4: 生成报告

**生成前先检查模板：** 如果 `templates/` 目录下存在 `custom-daily.md` 或 `custom-weekly.md`，**必须使用用户自定义模板**，不要用默认 SMART 模板。自定义模板优先级高于一切默认格式。

### 日报模板（默认 SMART，无自定义模板时使用）

```
## 📋 日报 — {YYYY年MM月DD日 星期X}

### 🎯 今日目标
> 今天要完成的 1-2 个核心目标（一句话说清楚）

### ✅ 完成情况

| 任务 | 产出/数据 | 进度 |
|------|----------|------|
| {具体任务1} | {量化结果，如：输出方案 2 份、代码提交 3 次} | ✅ 完成 |
| {具体任务2} | {量化结果} | 🔄 {N}% |
| {具体任务3} | {量化结果} | ✅ 完成 |

### ⚠️ 阻塞与问题
- **{问题描述}** — 影响：{简述} — 需要：{谁来协助/什么资源}
- （无则写"无"）

### 🔜 明日计划

| 事项 | 预估耗时 | 截止时间 |
|------|---------|----------|
| {任务1} | {N}h | 明天下班前 |
| {任务2} | {N}h | 明天 15:00 |

### 💡 今日小结
> 一句话总结今天的核心进展 + 一个关键收获/教训
```

### 周报模板（完整 SMART）

```
## 📊 周报 — {YYYY年MM月DD日 - MM月DD日}

### 🎯 本周目标回顾
| 目标（SMART） | 完成度 | 状态 |
|---------------|--------|------|
| {目标1：具体+量化+时限} | {N}% | 🟢 达标 |
| {目标2} | {N}% | 🟡 滞后 |

### ✅ 本周完成

**{重点项目1}** — 关联目标：{对应哪个大目标}
- 做了什么：{具体行动}
- 产出数据：{量化结果，如 DAU +5%、响应耗时降低 200ms}
- 关键节点：{哪天完成的 / 哪天上的线}

**{重点项目2}**
- ...

### ❌ 未完成 & 根因分析

| 事项 | 原因 | 解决方案 | 需要支持 |
|------|------|---------|---------|
| {事项} | {根因，非借口} | {补救措施} | {如需跨团队协调} |

### 📊 关键数据看板

| 指标 | 当前值 | 目标值 | 趋势 |
|------|--------|--------|------|
| {指标1} | {数值} | {目标} | ↑ ↓ → |
| {指标2} | {数值} | {目标} | ↑ ↓ → |

### ⚠️ 风险与决策请求

| 风险/议题 | 影响 | 建议方案 | 需谁决策 |
|-----------|------|---------|---------|
| {描述} | {影响评估} | {A/B 选项对比} | {决策人} |

### 📋 下周计划

| 事项（SMART） | 优先级 | 预估耗时 | 截止日期 |
|---------------|--------|---------|----------|
| {具体任务+量化目标} | P0/P1/P2 | {N}h | MM-DD |
| {具体任务+量化目标} | P0/P1/P2 | {N}h | MM-DD |

### 💡 本周复盘
> 1. **做对了什么**：{经验/方法沉淀}
> 2. **做错了什么**：{教训/改进点}
> 3. **下周最该改的一件事**：{1 条 actionable 的行动项}
```

## Phase 5: 输出

### 默认：聊天框直接展示

直接输出格式化好的报告，用户可复制粘贴。

### 选项：自动填写回公司系统

如果用户之前是通过「自动登录系统拉取」(Phase 2B) 获取的数据，生成报告后可以**直接填写回系统**：

```bash
# 保存日报（仅草稿，不提交）
python3 scripts/save_report.py <username> <password> \
  --daily "<日报内容>" --date 2026-06-03 --status 0

# 保存周报（仅草稿，不提交）
python3 scripts/save_report.py <username> <password> \
  --weekly "<周报内容>" --week-start 2026-06-01 --week-end 2026-06-07 --status 0
```

> **status 参数：**
> - `0` = 仅保存草稿，不提交（**默认**，用户后续可登录系统修改）
> - `1` = 直接提交

**工作流程：**
1. AI 按 SMART 模板生成报告内容
2. 将内容作为 `--daily` 或 `--weekly` 参数传入脚本
3. 脚本登录 → 调 `POST /api/statement/daily` 或 `POST /api/statement/week` → 保存
4. 如果报告已存在（code=006），自动降级为 PUT 更新

**保存前必须询问用户确认**，展示完整报告内容，让用户审核后再写入系统。

## AI 写作规则（SMART 强制执行）

生成报告时 AI 必须遵守：

1. **拒绝流水账**：不写"做了 A、做了 B、做了 C"这种纯列表，每条工作必须带量化数据或产出物说明
2. **量化优先**：能用数字就不用形容词。"优化了性能" → 「接口耗时从 800ms 降到 200ms（↓75%）」
3. **暴露阻塞**：日报中的「阻塞与问题」不能为空或敷衍。如果用户没说，主动问一句"今天有遇到什么卡点吗？"
4. **区分状态**：
   | 状态 | 图标 | 含义 |
   |------|------|------|
   | 完成 | ✅ | 已交付/已上线 |
   | 进行中 | 🔄 | 附进度百分比 |
   | 滞后 | 🟡 | 偏离计划，标注原因 |
   | 阻塞 | 🔴 | 需外部介入 |
5. **根因不甩锅**：未完成事项的原因分析要挖到根因，不是借口。❌「时间不够」→ ✅「需求评审晚了 2 天，压缩了开发时间」
6. **下周计划必须 SMART**：不能写「继续推进 XX」，要写「完成 XX 模块的 API 开发（3 个接口），预计周三提测」
7. **复盘要有行动项**：周报的"本周复盘"最后一条必须是下周立刻能做的一件事，不能是空洞的"继续努力"
8. **空数据兜底**：无法获取工作数据时，明确告知用户，并提供模板让用户手动填写
